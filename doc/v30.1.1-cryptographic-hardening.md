# v30.1.1 cryptographic hardening record

This document records the v30.1.1 cryptographic implementation, failure
semantics, provenance checks, and validation budgets. It is an audit record,
not a substitute for the release gates in `doc/v30.1.1-release-gate.md`.

## Implementations and provenance

The release uses ML-DSA-44 from liboqs 0.15.0 and the reference Argon2id
implementation from the Password Hashing Competition's `20190702` release.
The exact tags, tag commits, archive URLs, archive SHA-256 hashes, relevant
upstream file hashes, vendored Argon2 file hashes, and documented local Argon2
deltas are in:

```
contrib/devtools/quantum-crypto-provenance.json
```

Verify a source checkout and downloaded upstream archives with:

```
contrib/devtools/verify-quantum-crypto-sources.py \
  --liboqs-archive /path/to/liboqs-0.15.0.tar.gz \
  --argon2-archive /path/to/argon2-20190702.tar.gz \
  --wycheproof-json /path/to/mldsa_44_verify_test.json
```

The mandatory safety gate performs the same verification from freshly
downloaded, digest-pinned objects and retains a machine-readable evidence
artifact. The manifest also pins the upstream liboqs RNG implementation and
interface. The verifier rejects production source that calls liboqs APIs which
replace or switch its randomness provider. Reproduce the exact evidence path
from a clean checkout with:

```
SOURCE_COMMIT=$(git rev-parse HEAD)
contrib/devtools/verify-quantum-crypto-sources.py \
  --fetch-upstream /tmp/blackcoin-quantum-crypto \
  --require-upstream \
  --repository Blackcoin-Dev/Blackcoin \
  --source-commit "$SOURCE_COMMIT" \
  --evidence /tmp/blackcoin-quantum-crypto-evidence.json
```

The evidence records `Blackcoin-Dev/Blackcoin` and the full 40-character
checkout commit internally. Generation fails if the repository identity,
checkout `HEAD`, or clean-worktree check does not match those values.

The ML-DSA known-answer data is generated from C2SP/Wycheproof commit
`6d7cccd0fcb1917368579adeeac10fe802f1b521`, whose source JSON has SHA-256
`5ec04790c240c443ca8b662b8fc871834602c7cce87fcd36a193110745b2b9ea`.
`contrib/devtools/gen-mldsa44-kat.py` checks those identifiers and sizes before
generating `src/crypto/mldsa_kat.h`. The mandatory provenance job regenerates
that header from the freshly downloaded and digest-verified JSON and requires
an exact byte-for-byte match with the checked-in header. The startup test
verifies both a valid ML-DSA-44 vector and a non-canonical-signature rejection
vector.

The independent Argon2id startup vector is RFC 9106 section 5.3: version 19,
three iterations, 32 KiB, four lanes, password `32*0x01`, salt `16*0x02`,
secret `8*0x03`, and associated data `12*0x04`. Its expected 32-byte tag is:

```
0d640df58d78766c08c037a34a8b53c9d01ef0452d75b65eb52520e96b01e659
```

Startup fails closed if the linked liboqs version or ML-DSA metadata and sizes
do not match the pinned implementation, or if either independent known-answer
test fails. Release binaries must use the pinned `depends` build. A passing
test against a host-installed liboqs is not proof of release-binary provenance.

## Canonical encodings

Consensus ML-DSA inputs are exact-width:

| Field | Required bytes |
| --- | ---: |
| ML-DSA-44 public key | 1,312 |
| ML-DSA-44 secret key (local signing only) | 2,560 |
| ML-DSA-44 signature | 2,420 |
| signed consensus digest | 32 |

Short, long, null, malformed, and non-canonical ML-DSA inputs are rejected.
The pinned reference verifier rejects repeated, unordered, or non-zero-padded
hint encodings. The wrapper never pads, truncates, or re-encodes consensus
inputs.

New shadow PoW work uses the byte-exact `QQP2` payload: four magic bytes, one
PoW mode byte, an eight-byte little-endian nonce, a two-byte little-endian
target length, the target script, a two-byte little-endian payout-script
length, and the payout script. Both lengths must be non-zero and must consume
the payload exactly. The payout must be a direct quantum-migration script. The
prefixed payload generated for an `OP_RETURN` is limited to 520 bytes. Existing
historical `QQP1` parsing is retained for historical-state compatibility; new
Gold Rush claims must be quantum-linked `QQP2` proofs.

## Local failure is not consensus invalidity

Cryptographic APIs distinguish three outcomes: valid/success, invalid input or
proof, and local internal failure. The last category includes an injected
allocation/library failure and is retryable. It must never:

- mark a transaction or block consensus-invalid;
- populate the full-script success or failure cache;
- evict an otherwise valid transaction during reorg reacceptance; or
- partially mutate demurrage or shadow state.

This rule is enforced at quantum script verification, quantum PoS block
signatures, demurrage attestations, shadow PoW proof validation, mempool and
package admission, parallel script-check completion, block connection, and
reorg reacceptance. Block-processing local failures abort the local operation;
the same bytes can be retried after the local condition clears.

ML-DSA operations share an immutable initialized liboqs descriptor. No startup
or known-answer test replaces liboqs's process-global random generator. This
removes the former race between self-test RNG replacement and concurrent key
generation. Test-only fault counters are atomic and have no configuration,
RPC, or network-input path.

## Validation budgets

### Shadow Argon2id

Each shadow proof evaluation uses Argon2id version 19 with `t=1`, `m=1024 KiB`,
and `p=1`, producing a 32-byte hash. Block validation performs these evaluations
serially and stops after 64 candidates. The resulting worst-case Argon2 budget
is 64 evaluations and approximately 64 MiB of memory-matrix traffic per block;
the current validation loop has approximately one 1 MiB matrix live at a time,
plus implementation overhead. One valid claim can be credited per block. The
benchmark calls the same public `SHADOW_ARGON2_TIME_COST`,
`SHADOW_ARGON2_MEMORY_KIB`, and `SHADOW_ARGON2_LANES` constants as production
validation; compile-time drift cannot silently leave the benchmark on stale
literal parameters.

On an Apple M5 arm64 development host (macOS 27.0, Apple clang 21, optimized
build), seven local batch samples measured:

| Operation | Minimum | Median | Maximum |
| --- | ---: | ---: | ---: |
| one `t=1, m=1024 KiB, p=1` Argon2id evaluation | 263.852 us | 267.263 us | 372.301 us |
| 64 Argon2id evaluations | 17.184 ms | 17.220 ms | 19.474 ms |

These figures are a development reference, not a release threshold. The safety
gate measures both `QuantumArgon2id1MiB` and the complete bounded
`QuantumArgon2id64ClaimBlock` path. The latter performs the production cap of
64 sequential 1 MiB evaluations and records a nanobench batch of exactly 64.
Both the single-operation and block benchmarks fail if Argon2 returns an error.
Sanitizer and slow-emulation jobs need separate timeout allowances.

### ML-DSA verification and the 32,000,000-weight-unit block limit

After V4 activation the maximum block weight is 32,000,000 weight units and the
maximum signature-operation cost is 640,000. One quantum ML-DSA witness spends
16 signature-operation units, so the signature-operation ceiling alone permits
40,000 such verifications. Weight is the tighter bound.

A quantum input has at least 41 non-witness bytes and a 3,739-byte witness stack:
one stack-count byte, a three-byte length and 2,420-byte signature, and a
three-byte length and 1,312-byte public key. Its minimum input weight is
`41*4 + 3739 = 3903`. Thus the block-weight-only upper bound is
`floor(32,000,000 / 3,903) = 8,198` quantum input verifications. The realizable
count is lower because every valid transaction and block also has version,
counts, outputs, locktime, and block overhead.

Demurrage attestations have independent hard caps of one per transaction and 16
per block. A quantum PoS block adds at most one block-signature verification.
The conservative sum of independently capped ML-DSA verifications is therefore
8,215 per block, although attestation and transaction bytes reduce the
realizable script-input count. The benchmark derives 3,903 at compile time from
the serialized outpoint, empty scriptSig length, sequence, witness item count,
compact-size signature and public-key lengths, the ML-DSA byte constants, and
the witness weight scale. Compile-time assertions then derive 8,198 and 8,215
from the production block-weight and demurrage limits.

On the same development host, seven local samples of an independent-vector
ML-DSA-44 verification measured 48.930 us minimum, 62.700 us median, and
71.017 us maximum. Multiplying the median by the conservative 8,215 upper bound
gives about 515 ms of serial verification work before cache/parallelism effects
and other block-validation costs. This is only a sizing reference. The safety
gate measures both `QuantumMLDSA44Verify` and
`QuantumMLDSA44MaxWeightBlock`; the block benchmark performs all 8,215
verifications rather than only multiplying one measurement and records a batch
of exactly 8,215. Both benchmarks fail if the independent vector does not
verify.

Linux x86-64 and native Intel/Apple-Silicon macOS jobs retain the raw nanobench
JSON and schema-validated evidence. The generator rejects missing, extra, or
duplicate benchmark names, changed units or batch counts, fewer than five
epochs, and median or total-time summaries that do not reproduce from those
epochs. Each evidence record binds the raw JSON and benchmark executable to the
exact `Blackcoin-Dev/Blackcoin` checkout `HEAD` and records native endianness,
actual configured `CXXFLAGS`, the named build profile, configured minimum
runtime, and the SHA-256 of the quantum-cryptography provenance manifest. The
evidence schema reserves separate
`large-block` and `synthetic-state` domains. Production mode deliberately fails
closed until a 32 MiB block-validation benchmark and maximum-marker
apply/undo benchmark populate those domains. The crypto resource job therefore
does not represent those two outstanding domains as complete.

## Required release evidence

Before production release, retain artifacts showing:

1. provenance verification against freshly downloaded upstream archives;
2. startup known-answer tests and the complete crypto hardening unit suite;
3. fault-injection retry tests for script, block-signature, demurrage, and
   shadow proof paths;
4. concurrency tests under ThreadSanitizer, plus ASan and UBSan runs;
5. the quantum crypto fuzz target on malformed and boundary-size inputs;
6. source-bound benchmark results from pinned `depends` builds on x86-64 and
   arm64, including the measured 64-Argon2 and 8,215-ML-DSA block bounds, plus
   dedicated 32 MiB block-validation and maximum synthetic-state apply/undo
   results; the supported Windows build and any big-endian release target must
   be covered by the platform matrix; and
7. reindex and historical lifecycle/reorg tests demonstrating unchanged
   historical results.

Local arm64 unit and concurrency results do not waive these release gates.
