# v30.1.1 competing Gold Rush PoW claim rule

## Activation and replay

The canonical competing-claim rule activates at mainnet height 5,993,200, the
first scheduled Gold Rush reward halving. This is the new v30.1.1 QQP3/rank-v1
boundary. Blocks through 5,993,199 retain the v30.1.0 rule: the first
proof-valid, input-matched claim in base-block transaction order receives the
fixed PoW pool and later valid claims receive no shadow credit. This preserves
already-mined v30.1.0 shadow history. From 5,993,200 onward, the
order-independent rule below applies. Neither rule changes legacy-visible
transactions or base-block validity.

QQP4 exact-input proof binding is a distinct consensus activation. Mainnet
v30.1.1 sets its activation height to `INT_MAX`, so QQP4 is
disabled. A readiness bit, version bit, or other signalling state cannot
activate it. A future QQP4 release must publish a separately reviewed height,
stay after the QQP3 boundary, and prove the transition for Q3 claims that are
still eligible for late inclusion.

The authenticated shadow replay-state schema is version 12 and commits to the
QQP3 and QQP4 schedule inputs. An upgrade from v30.1.0 therefore requires the
documented full `-reindex-chainstate` replay, but that replay reproduces
v30.1.0 monetary results before 5,993,200 and changes canonical accounting
only at the announced QQP3 boundary.

Every wallet, staking or mining node, explorer, and indexer that consumes
shadow state must upgrade before height 5,993,200. The two versions can accept
the same Gold Rush base blocks while intentionally deriving different shadow
recipients and balances. A wallet/datadir that has processed post-boundary
v30.1.1 state must not be reopened in v30.1.0; rollback uses the cold
pre-upgrade copy. This distinction is why base-chain compatibility is not a
claim of identical shadow-ledger state.

## Canonical evaluation

Wallet-created proofs remain byte-for-byte `QQP2` before height 5,993,200.
From the QQP3 boundary while QQP4 remains disabled, the wallet emits `QQP3`,
which adds the intended block height and that block's parent hash. The new
v30.1.1 QQP3 validation, canonical rank-v1 accounting, and late-origin behavior
take effect at that boundary. A QQP3 proof is eligible in its intended
block or for 64 later blocks only when its parent remains the active-chain
ancestor. An old `QQP2` omitted before an origin-bound rule has no
authenticated origin and cannot be reimbursed retroactively; the wallet keeps
its input quarantined until its own confirmation or a confirmed on-chain
conflict. Generic `abandontransaction` intentionally refuses that claim while
a peer can still confirm it.

The wallet provides one consent-only guided conflict path for an exact
persisted, wallet-authored, single-input quarantined claim.
`createshadowpowclaimresolution <claim_txid>` is a dry run by default and
returns the exact input, amount, fee, same-script output, and maximum virtual
size without signing. `dry_run=false` requires a normally unlocked wallet and
`acknowledge_fee_and_conflict_risk=true`; it returns signed hex but never
broadcasts. The operator must review it and call `sendrawtransaction`
separately. The RPC refuses live claims, unresolved descendants or conflicts,
spent or foreign inputs, incomplete sync, unsafe fees, staking-only unlock,
and disabled private keys. A confirmed resolution pays an unreimbursed base
fee; peers retaining the original may reject it, confirmation is not
guaranteed, and the wallet reserves the shared input until one side confirms.
After an explicit broadcast and wallet recognition, ordinary wallet rebroadcast
may continue across restart as a consequence of that prior decision.

QQP4 adds the exact single legacy fee-input outpoint. It is rejected until its
separate consensus activation. If a future QQP4 activation is proposed, the
specification and tests must state whether a QQP3 claim already inside its
64-block late-inclusion window is honored, expires, or is otherwise handled at
the boundary; v30.1.1 makes no mainnet claim about that future
transition.

A regular transaction in a Proof-of-Stake block may carry exactly one
`QQSPROOF` output. Coinbase, coinstake, and Proof-of-Work-block locations are
ineligible. A transaction with multiple `QQSPROOF` outputs is malformed for
shadow accounting. Neither an ineligible nor malformed note receives a
credit or fee reimbursement.

Every eligible proof transaction receives this canonical rank:

```
logical_proof_id = Hash("Quantum Quasar Logical POW Proof Identity v1",
                        decoded_version,
                        decoded_mode,
                        decoded_nonce,
                        decoded_target_script,
                        decoded_quantum_payout_script,
                        decoded_origin,
                        decoded_bound_input_if_present)

Hash("Quantum Quasar Canonical POW Logical Proof Rank v1",
     height,
     previous_block_hash,
     logical_proof_id)
```

The proof decoder consumes every length-delimited field and rejects trailing
bytes. The logical ID therefore commits to the canonical decoded QQP2/QQP3
tuple, not to the transaction ID, output number, or outer `OP_RETURN` push
opcode. Non-minimal and minimal outer pushes of the same payload are the same
logical proof. Different nonces, target scripts, payout scripts, origins, or
future bound inputs remain different proofs.

Each logical proof receives one rank and at most one Argon2id evaluation per
block. Additional carrier transactions containing that proof increment
`duplicate_logical_proof_count` and cannot receive another row, winner chance,
or reimbursement. After proof validation, the representative carrier is the
eligible transaction with the greatest `min(actual_fee, 0.01 BLK)`; ties use
the smallest txid and then output number. A zero-fee copy therefore cannot
suppress a legitimate reimbursement. The current block hash is not an input,
so merely permuting the same transactions cannot change the result. This is
the QQP3 canonical rank-v1 construction. QQP4 uses a distinct rank-v2
construction only after its separately scheduled activation.

Validation performs at most 64 expensive Argon2id evaluations per block. The
block is scanned without retaining a list of its notes or transactions. A
fixed-size top-64 selector retains only the lowest canonical logical proofs;
structurally invalid notes update fixed counters and do not consume an Argon2
evaluation. Every remaining eligible candidate is represented by the
aggregate `evaluation_limit_count` and receives nothing. This produces the
same winner and reimbursements as sorting the complete candidate set without
the complete set's memory cost.

Mempool and block-template policy retain the historical one-proof limit before
the QQP3 boundary and use the same 64-proof cap thereafter. A QQP3
origin-bound proof remains relayable while its origin is on the active branch
and its age is at most 64 blocks. Connect and disconnect revalidate those
conditions instead of deleting every proof unconditionally. A future QQP4
activation must add its own boundary-transition tests rather than reusing this
statement as an activation rule.

From the QQP3 boundary onward, mempool and package policy also reject a second
carrier of a logical proof already in the mempool. An authenticated per-block
bucket records every winner, current loser, and accepted late proof, including
a valid zero-fee loser. The evaluator scans only the preceding 64 active-branch
buckets, so the same solution cannot receive another reimbursement in a later
eligible block. Every active reward block at and after the QQP3 boundary writes
an authenticated bucket, including an empty one. Disconnect removes that
branch's bucket, restart verifies the complete window through the replay-state
commitment, and reindex reconstructs it from base blocks. A missing, malformed,
off-branch, or repeated bucket entry is a local replay error and never makes
the shared base block invalid. The finite reward epoch retains its bucket
history so an ordinary deep reorg can authenticate and undo each disconnected
height; it does not create Migration or Final buckets.

## Fixed-pool monetary rule

The lowest-ranked evaluated valid claim whose origin is the current block is
the winner. Each other valid current-origin claim and every eligible late-origin
claim receives its actual base-chain transaction fee, capped at 0.01 BLK. If a
current winner exists, it receives the fixed PoW pool minus all reimbursements.
If a block contains only late claims, it pays only those reimbursements and
leaves the rest of the pool accumulated; a late proof never wins the jackpot,
resets PoW difficulty, or increments the PoW solve count. Invalid, malformed,
ineligible, input-mismatched, expired, origin-mismatched, duplicate,
already-accounted, and over-limit claims receive nothing.

The actual base fee is reconstructed from the connected block undo data:

```
actual base fee = sum(spent base UTXO values) - sum(transaction outputs)
claim reimbursement = min(actual base fee, 0.01 BLK)
winner credit = fixed PoW pool - sum(current-loser and late reimbursements)
```

At most 63 claims can be reimbursed alongside one current winner, so the
maximum aggregate reimbursement in a winner block is 0.63 BLK. The smallest
production Gold Rush PoW credit is 231.5 BLK
(`463 BLK / 2`). The reimbursements therefore always fit, the winner always
receives a positive remainder, and the sum of all PoW credits is exactly the
pre-existing fixed pool. The rule neither mints an additional satoshi nor
converts a claimant's fee into base-chain subsidy.

An attacker-selected future, expired, or off-branch origin is an ordinary
invalid claim and cannot fail-stop a block. Once the committed height and
parent match an active-chain ancestor, however, that ancestor's authenticated
`QQPUND` marker is mandatory. Missing or malformed authenticated undo data,
arithmetic overflow, and local Argon2 failure are local internal errors. The
node stops shadow-state application without marking the otherwise
legacy-valid base block invalid.

Shadow application is transactional. All authenticated shadow mutations for a
base block are staged in a child coins cache and published only after the
complete proof/accounting pass succeeds. The caller's outer block cache is
discarded on any local error, so neither partial credits nor an advanced
shadow marker can reach durable chainstate. Retrying or reindexing the same
block derives the same winner, reimbursements, pool remainder, and synthetic
transaction identifiers.

Publishing the child into its parent cache is not independently
allocation-atomic. Production safety relies on `ConnectBlock` receiving a
transaction-local outer cache that its caller discards on every failure. The
production connection, validity-test, and database-verification callers follow
that contract; no production path applies shadow state directly to the durable
coins tip.

## Explorer contract

`PrepareShadowPowClaimAccounting` copies a small immutable context from the
authenticated historical pool state without performing Argon2 work. An index
can call it while holding its chain/view lock, release that lock, and pass the
context to `EvaluateShadowPowClaimAccounting`. The evaluator reads only the
immutable context, block, and block undo. `GetShadowPowClaimAccounting` is a
convenience wrapper for callers that do not need the split. The evaluator is
the same bounded engine used by `ApplyShadowBlock`.
It clears its result vector and returns no partial classification prefix when
allocation, Argon2, or authenticated-undo evaluation fails. `shadowindex`
performs that evaluation before constructing or writing its atomic database
batch. A failed index append does not advance the index locator; restart
retries the block.

Detailed rows are returned only for the at most 64 canonically selected
logical proofs. Each row contains the source txid, source vout, logical proof
ID, canonical rank, decoded quantum payout when available, actual fee when
known, credited amount, and one stable disposition:

- `INVALID_PROOF`
- `INPUT_MISMATCH`
- `INVALID_BASE_FEE`
- `WINNER`
- `REIMBURSED_LOSER`
- `ORIGIN_MISMATCH`
- `ORIGIN_EXPIRED`
- `REIMBURSED_LATE`
- `ALREADY_ACCOUNTED`

All notes, including invalid locations, malformed multi-note transactions,
wrong/unknown modes, duplicate logical-proof carriers, and candidates outside
the evaluation budget, are counted
in deterministic bounded aggregates. `accounting_commitment` hashes the exact
ordered note stream, immutable accounting context, complete aggregate vector,
and every retained detail row. An explorer can recompute it from the anchored
base block and undo/context data without trusting omitted per-note rows.

Every detail row stores its `proof_version`, whether it is origin-bound, origin
height and parent hash, inclusion height, origin age, whether it is input-bound,
and the exact `claim_outpoint` when bound. `shadowindex` schema 11 enforces
`record_count == evaluated_count <= 64` and
stores only those detail rows. Its observed transaction-id array is a bounded
base-block-order sample of at most 64 entries. Undo, reconnect, restart, and
reindex therefore create a constant-sized claim batch even for a maximum-weight
legacy-valid block filled with malformed notes.

The helper accepts either the exact pre-application view or a later active-tip
view that still contains the authenticated per-block pool-undo marker. In the
latter case it reconstructs the exact historical pre-pool from that marker; it
never substitutes the current tip's pool. The connected base block undo is
still required for source inputs and actual fees. An index must not infer
winner or reimbursement status from transaction order or payout amount.

`shadowindex` schema 11 stores at most 64 evaluated classifications under
block-scoped ordinal keys rather than embedding an attacker-sized vector in
the block record. `getshadowblock` schema `blackcoin.shadow.block.v3` pages
those records independently from synthetic payouts and exposes fixed outcome
aggregates, exact retained fee and credit provenance, and the complete-stream
commitment. Rewind deletes the bounded block-scoped records in the same atomic
batch as the block summary; reindex rederives them through this shared engine.

Schema 11 invalidates earlier prerelease explorer records created with the
superseded activation, incomplete proof-mode/spend-anchor data, or unbounded
per-note accounting. Schema 11 adds carrier-independent logical proof IDs and
duplicate/already-accounted counters. Coinstatsindex schema 3 likewise
invalidates prerelease schema-2 MuHash and per-block payout records. Both
optional indexes recognize the older schema, wipe only their reconstructible
index database, and rebuild from local active-chain block data. A
`-reindex-chainstate` alone does not wipe optional indexes, so this independent
schema invalidation is required.

## Required regression evidence

- transaction-order permutations choose the same winner and amounts;
- actual fees below 0.01 BLK are reimbursed exactly;
- fees above 0.01 BLK are capped;
- malformed, invalid, input-mismatched, Proof-of-Work-block, and excess notes
  receive zero;
- 65 valid candidates produce one winner, 63 reimbursements, and an aggregate
  `evaluation_limit_count` of one without a 65th detail row;
- all credited amounts sum exactly to the fixed pool;
- a late-only block pays only capped fees, preserves the pool remainder, and
  does not record a current solve;
- a mixed current/late block preserves global canonical row/payout order and
  links every synthetic payout to the correct source;
- age 64 is eligible, age 65 is expired, and off-branch origins are nonfatal;
- minimal and non-minimal carriers of one decoded proof produce one rank, one
  Argon2 evaluation, and at most one reimbursement, while different valid
  nonces remain independent candidates;
- the highest eligible capped-fee carrier is selected deterministically and a
  zero-fee duplicate cannot suppress reimbursement;
- a proof accounted in one block is rejected throughout its remaining
  64-block window, while disconnecting that block removes its branch bucket;
- a matching active origin with missing/corrupt authenticated undo fails
  closed;
- disconnect/reconnect, alternate-block reorg, restart, and replay reproduce
  the same accounting; and
- local cryptographic, allocation, or undo-data failures do not poison
  base-block validity, publish partial shadow state, or expose partial index
  classifications.
