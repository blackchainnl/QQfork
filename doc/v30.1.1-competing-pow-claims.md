# v30.1.1 competing Gold Rush PoW claim rule

## Activation and replay

The canonical competing-claim rule activates at mainnet height 5,993,200, the
first scheduled Gold Rush reward halving. Blocks through 5,993,199 retain the
v30.1.0 rule: the first proof-valid, input-matched claim in base-block
transaction order receives the fixed PoW pool and later valid claims receive no
shadow credit. This preserves already-mined v30.1.0 shadow history. From
5,993,200 onward, the order-independent rule below applies. Neither rule changes
legacy-visible transactions or base-block validity.

The authenticated shadow replay-state schema is version 11 and commits to the
activation height. An upgrade from v30.1.0 therefore requires the documented
full `-reindex-chainstate` replay, but that replay reproduces v30.1.0 monetary
results before 5,993,200 and changes accounting only at the announced future
boundary.

Every wallet, staking or mining node, explorer, and indexer that consumes
shadow state must upgrade before height 5,993,200. The two versions can accept
the same Gold Rush base blocks while intentionally deriving different shadow
recipients and balances. A wallet/datadir that has processed post-boundary
v30.1.1 state must not be reopened in v30.1.0; rollback uses the cold
pre-upgrade copy. This distinction is why base-chain compatibility is not a
claim of identical shadow-ledger state.

## Canonical evaluation

A regular transaction in a Proof-of-Stake block may carry exactly one
`QQSPROOF` output. Coinbase, coinstake, and Proof-of-Work-block locations are
ineligible. A transaction with multiple `QQSPROOF` outputs is malformed for
shadow accounting. Neither an ineligible nor malformed note receives a
credit or fee reimbursement.

Every eligible proof transaction receives this canonical rank:

```
Hash("Quantum Quasar Canonical POW Claim Rank v1",
     height,
     previous_block_hash,
     source_txid,
     source_vout,
     exact_proof_payload)
```

Candidates are ordered by rank, then source txid, source vout, and exact proof
payload as deterministic collision tie-breakers. The current block hash is not
an input, so merely permuting the same transactions cannot change the result.

Validation performs at most 64 expensive Argon2id evaluations per block. The
evaluated set is selected in canonical order. Structurally invalid notes do not
consume an Argon2 evaluation. Once the bound is reached, every remaining
eligible candidate is classified `EVALUATION_LIMIT` and receives nothing.

## Fixed-pool monetary rule

The lowest-ranked evaluated valid claim is the winner. Each other evaluated
valid claim receives its actual base-chain transaction fee, capped at
0.01 BLK. The winner receives the fixed PoW pool minus those reimbursements.
Invalid, malformed, ineligible, input-mismatched, and over-limit claims receive
nothing.

The actual base fee is reconstructed from the connected block undo data:

```
actual base fee = sum(spent base UTXO values) - sum(transaction outputs)
loser reimbursement = min(actual base fee, 0.01 BLK)
winner credit = fixed PoW pool - sum(loser reimbursements)
```

At most 63 losers can be reimbursed, so the maximum aggregate reimbursement is
0.63 BLK. The smallest production Gold Rush PoW credit is 231.5 BLK
(`463 BLK / 2`). The reimbursements therefore always fit, the winner always
receives a positive remainder, and the sum of all PoW credits is exactly the
pre-existing fixed pool. The rule neither mints an additional satoshi nor
converts a claimant's fee into base-chain subsidy.

Missing or malformed local undo data, arithmetic overflow, and local Argon2
failure are local internal errors. The node stops shadow-state application
without marking the otherwise legacy-valid base block invalid.

Shadow application is transactional. All authenticated shadow mutations for a
base block are staged in a child coins cache and published only after the
complete proof/accounting pass succeeds. The caller's outer block cache is
discarded on any local error, so neither partial credits nor an advanced
shadow marker can reach durable chainstate. Retrying or reindexing the same
block derives the same winner, reimbursements, pool remainder, and synthetic
transaction identifiers.

Publishing the child into its parent cache is not independently
allocation-atomic. Production safety relies on `ConnectBlock` receiving a
transaction-local outer cache that its caller discards on every failure. The
production connection, validity-test, and database-verification callers follow
that contract; no production path applies shadow state directly to the durable
coins tip.

## Explorer contract

`PrepareShadowPowClaimAccounting` copies a small immutable context from the
authenticated historical pool state without performing Argon2 work. An index
can call it while holding its chain/view lock, release that lock, and pass the
context to `EvaluateShadowPowClaimAccounting`. The evaluator reads only the
immutable context, block, and block undo. `GetShadowPowClaimAccounting` is a
convenience wrapper for callers that do not need the split. The evaluator is
the same bounded engine used by `ApplyShadowBlock`.
It clears its result vector and returns no partial classification prefix when
allocation, Argon2, or authenticated-undo evaluation fails. `shadowindex`
performs that evaluation before constructing or writing its atomic database
batch. A failed index append does not advance the index locator; restart
retries the block.

For every observed proof note it returns the source txid, source vout,
canonical rank, decoded quantum payout when available, actual fee when known,
credited amount, and one stable disposition:

- `INVALID_LOCATION`
- `MALFORMED_TRANSACTION`
- `INVALID_PROOF`
- `WRONG_MODE`
- `UNKNOWN_MODE`
- `INPUT_MISMATCH`
- `INVALID_BASE_FEE`
- `EVALUATION_LIMIT`
- `WINNER`
- `REIMBURSED_LOSER`

The helper accepts either the exact pre-application view or a later active-tip
view that still contains the authenticated per-block pool-undo marker. In the
latter case it reconstructs the exact historical pre-pool from that marker; it
never substitutes the current tip's pool. The connected base block undo is
still required for source inputs and actual fees. An index must not infer
winner or reimbursement status from transaction order or payout amount.

`shadowindex` schema 6 stores each classification under a block-scoped ordinal
key rather than embedding an attacker-sized vector in the block record.
`getshadowblock` schema `blackcoin.shadow.block.v2` pages those records
independently from synthetic payouts and exposes exact winner, loser,
rejection, fee, credit, and synthetic-output provenance. Rewind deletes the
block-scoped records in the same atomic batch as the block summary; reindex
rederives them through this shared engine.

Schema 6 invalidates prerelease schema-4 explorer records created with the
superseded height-5,950,000 activation and schema-5 records that lacked exact
wrong/unknown proof-mode dispositions. Coinstatsindex schema 3 likewise
invalidates prerelease schema-2 MuHash and per-block payout records. Both
optional indexes recognize the older schema, wipe only their reconstructible
index database, and rebuild from local active-chain block data. A
`-reindex-chainstate` alone does not wipe optional indexes, so this independent
schema invalidation is required.

## Required regression evidence

- transaction-order permutations choose the same winner and amounts;
- actual fees below 0.01 BLK are reimbursed exactly;
- fees above 0.01 BLK are capped;
- malformed, invalid, input-mismatched, Proof-of-Work-block, and excess notes
  receive zero;
- 65 valid candidates produce one winner, 63 reimbursements, and one
  `EVALUATION_LIMIT` result;
- all credited amounts sum exactly to the fixed pool;
- disconnect/reconnect, alternate-block reorg, restart, and replay reproduce
  the same accounting; and
- local cryptographic, allocation, or undo-data failures do not poison
  base-block validity, publish partial shadow state, or expose partial index
  classifications.
