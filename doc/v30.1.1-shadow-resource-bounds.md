# v30.1.1 shadow-state resource bounds

This note separates deterministic protocol bounds from candidate observations
and from release benchmark evidence. The 5,993,200 competing-claim boundary is
a proposed v30.1.1 shadow-state rule, not a rule deployed by v30.1.0. The two
versions can accept the same base blocks while deriving different shadow state
from that boundary.

## Deterministic bounds

The legacy absolute claim-marker bound is derived from the maximum v4 block
weight and the minimum transaction weight:

```
(32,000,000 / 240) + 2 = 133,335 claims per block
```

That bound remains in force for already-mined v30.1.0 history and the
v30.1.1 candidate's pre-boundary replay. From the proposed v30.1.1 QQP3
competing-claim activation height onward, the tighter authenticated bound is:

```
min(133,335, authenticated whitelist entries + 64 PoW evaluations)
```

The PoS term does not create a new participant limit. The candidate's
active-signal state contains no more entries than the authenticated whitelist,
and payout-script merging can only reduce the number of PoS claim records. The
PoW term is the candidate's fixed validation cap. This is the resource bound
for the proposed post-boundary shadow rule; it does not describe v30.1.0
shadow allocation.

Every claim creates one authenticated claim record, one positive payout coin,
and one payout-provenance record. It also inserts one leaf into each of the
issued and unspent MuHash inventories. At the observed mainnet whitelist size
of 687, the post-activation ceiling is therefore 751 claims, 2,253 persistent
claim-family coins, and 1,502 MuHash insertions in a qualifying block. Fixed
pool, inventory, active-state, and replay records are separately bounded and
do not multiply by arbitrary user output count.

Claim enumeration no longer probes deterministic outpoints until the first
missing coin. It reads the exact count from the block's reserved `QQPUND`
record, validates that record's height, time, block hash, previous block hash
when branch context is available, post-activation whitelist bound, and claim
set hash, and then visits exactly that many records. A claim-shaped coin at the
next deterministic index cannot extend the authenticated set.

`getcirculatingsupply` does not materialize either the completed-epoch payout
inventory or the demurrage latest-state inventory in process memory. It uses
three independently owned immutable chainstate cursors: one sequential marker
cursor, one sequential monetary-output cursor, and one deterministic
latest-attestation cursor. Payout provenance and demurrage latest state are
authenticated by exact reserved-outpoint seeks against the same flushed tip.
A missing record is distinct from a present but corrupt record; corruption
fails the RPC instead of silently changing supply classification. The unit
suite performs 10,000 distinct authenticated attestation seeks and 10,000
repeated payout seeks without increasing cache memory after snapshot creation.

Maintenance no longer compares each legacy marker candidate against 133,335
possible indices. It constructs the expected outpoint map once from the
authenticated per-block claim count. If old state has no authentic pool undo,
the fallback is bounded by the number of candidates actually found. Legacy
active-signal cleanup uses the authenticated whitelist count. Marker shape by
itself never authorizes deletion.

Post-boundary `QQSPROOF` accounting has a separate count and storage bound.
Validation scans block outputs in place and retains only the lowest 64
canonical candidates. The conservative observation-counter ceiling is derived
from the absolute v4 block-weight limit and the nine-byte minimum serialized
output envelope:

```
32,000,000 / (4 * 9) = 888,888 observed notes
```

The Gold Rush contextual block limit is smaller, but the implementation uses
the absolute ceiling so the counter remains safe under every accepted block
context. Structural rejects and candidates outside the 64-proof budget update
fixed counters; they never allocate per-note accounting rows. Retained proof
payload bytes are also bounded by the serialized block that contains them.

Shadowindex schema 10 stores at most 64 detailed claim rows and at most 64
compatibility-sample transaction IDs per block. Its block summary contains a
fixed disposition vector and a commitment to the exact ordered note stream,
accounting context, aggregate vector, and retained rows. The claim RPC rejects
page sizes above 64. Disconnect erases exactly `record_count` detail keys, so
undo work cannot be amplified by omitted notes.

## State size and retention

The active-signal serializer is sharded at 8,000 payload bytes. Its allocation
limit is now derived from the authenticated whitelist manifest instead of the
generic 32 MiB chain-data deserialization limit. If the whitelist blob has
`W` bytes and `N` entries, the maximum active-state payload is
`min(33,553,408, W + 32 + 40N)` and the maximum undo payload is
`min(33,553,408, W + 169 + 41N)`. These expressions follow directly from the
whitelist, active-state, and undo encodings. They do not cap participants or
change a payout.

The 687-entry canonical-P2PKH fixture has an 18,554-byte whitelist blob. Its
resulting limits are exactly 46,066 state bytes and 46,890 undo bytes, each in
six shards. Reads reject a manifest above the authenticated limit before
reserving or reading the claimed shard payload. Writes and disconnect restores
use the same limit. The generic protocol envelope remains 33,553,408 bytes and
4,195 shards for a whitelist whose authenticated scripts actually require it.

| State family | Retention rule | Reason |
| --- | --- | --- |
| Current active-signal set and shards | Replace the prior snapshot immediately | Only the authenticated current set is authoritative |
| Active-signal undo and shards | Retain per state-changing block | Required for exact disconnect and reorg restoration |
| Pool, inventory, and replay tip records | Replace at the fixed reserved outpoint | They are rolling authenticated checkpoints |
| Per-block pool undo, claims, payout coins, and payout provenance | Retain through the finite Gold Rush history | Required for payout authentication, wallet/index reconstruction, audit, and reorg |
| Solver records | Finite Gold Rush records; direct recent lookups are used | Consensus-facing paths do not perform a full UTXO scan |
| QQPROOFS logical-proof buckets | One per height from 5,993,200 through 6,192,999; retain for the finite epoch | The active read window is 64 buckets, but arbitrary disconnect, reorg, and reindex require the authenticated per-block predecessor record |
| Spent-payout tombstones | One per actual base transaction input | Required to distinguish an authenticated spend from missing/corrupt payout state |
| Marker-shaped ordinary outputs | Never garbage-collected by shape | User outputs are preserved unless their reserved outpoint and provenance authenticate |

Normal startup uses the exact-tip replay checkpoint and does not perform a
full solver or claim inventory scan. It reads the current active-state shards:
six for the canonical-P2PKH fixture and at most 4,195 under the generic marker
envelope. The deep inventory pass remains an explicit maintenance/audit
operation. This bounds application-level startup reads, not LevelDB manifest
open time or wall-clock startup latency.

The v30.1.1 candidate's full height schedule has two allocation intervals.
Heights 5,950,000 through 5,993,199 contain 43,200 v30.1.0-compatible
allocation blocks with at most 687 PoS claims and one PoW claim. Heights
5,993,200 through 6,192,999 contain 199,800 new QQP3
canonical-reimbursement blocks with at most 687 PoS and 64 PoW claims. The
result is a full-epoch 687-entry candidate fixture count of 179,771,400 claims,
539,314,200 claim-family records, and 359,542,800 MuHash insertions. This is
tighter than multiplying the post-boundary 751-claim limit across historical
blocks where that rule is intentionally inactive.

The evidence generator also models the exact CoinEntry and CDBBatch encodings.
One maximum claim family is 509 batch-payload bytes: 205 for the authenticated
claim, 89 for its payout coin, and 215 for payout provenance. Pool undo and a
solver marker add at most 316 and 92 bytes per reward block. If every block
also retains the 46,890-byte canonical-P2PKH undo fixture, those append-only
families occupy 103,192,914,600 batch-payload bytes (96.11 GiB). The
QQPROOFS logical-proof bucket adds one retained zero-value Coin at each of the
199,800 post-boundary heights. Its exact batch-payload sizes are 98 bytes with
no proof IDs, 130 bytes with one ID, and 2,150 bytes at the 64-ID maximum.
Using the maximum at every height adds 429,570,000 bytes (409.67 MiB), for a
canonical-fixture total of 103,622,484,600 bytes (96.51 GiB). These per-entry
figures include the CoinEntry key and CDBBatch record framing, but exclude the
WriteBatch's shared 12-byte header. The source-only protocol envelope permits
133,335 claims per block, or 32,400,405,000 over the epoch.
Combining that count with the generic 33,553,408-byte undo envelope gives
24,754,927,050,000 retained append-only batch-payload bytes (22.51 TiB).

Neither number is a physical disk guarantee. They exclude the pre-existing
UTXO set, rolling checkpoint puts and deletes, LevelDB SSTable, index,
Bloom-filter and WAL overhead, obsolete files, and compaction amplification.
The contracted schedule, serializer, cache-allocation, and state-transition
sources are semantically checked and hashed into each resource evidence
artifact. The 2.0-second one-block apply/undo
qualification threshold sums to 486,000 seconds (135 hours) across 243,000
blocks, but that sum is not
a full replay wall-clock bound because it excludes base-block validation and
durable database I/O.

Deleting per-block authentication records earlier would weaken reorg, wallet,
and payout provenance and is not part of v30.1.1. Authenticated compaction is
not yet shown to be necessary, and it is not shown to be unnecessary either.
Issue #13 therefore remains Partial. The paired scoped synthetic and
current-live physical artifact is required evidence, but it does not establish
the future terminal combined-chainstate or full-cardinality production-RPC
bound. Those remain open until terminal-live evidence or a separately reviewed
conservative background model and, if required, authenticated compaction exist.

### Why a current-tip overlay is not terminal authorization

A protected copy of the current mainnet chainstate can be enlarged with
production-serialized Coins and can be used to stress the exact candidate
`getcirculatingsupply` implementation. That is useful supplemental evidence,
but it is not a terminal-chain equivalent. Future pool-undo, active-state, and
`QQPROOFS` records authenticate against the block hash, height, and time in the
active `CBlockIndex` chain. Records invented at the current tip cannot prove
that future authentication path. Setting the CoinsDB best block to an invented
terminal hash instead makes normal startup fail because the block index has no
such tip. A real reindex or reorg also reconstructs auxiliary state only from
actual blocks, so it cannot preserve an injected future history.

The ordinary UTXO background has no practical pre-terminal consensus maximum.
Ignoring transaction overhead, the 32,000,000-weight block limit can contain
at most `floor(32,000,000 / (4 * 9)) = 888,888` minimum serialized output
envelopes per block. Across all 243,000 Gold Rush heights, that gross ceiling is
215,999,784,000 outputs. A direct v16 quantum output occupies at least 43
serialized bytes, so the corresponding gross ceiling is
`floor(32,000,000 / (4 * 43)) = 186,046` per block, or 45,209,178,000 possible
latest-attestation seeks across the epoch. Transaction framing and inputs make
actual counts smaller, but no affordable current-tip multiplier is a
consensus-derived upper bound. This inherited base-chain fanout envelope is
separate from the bounded shadow claim families above.

Accordingly, a current-tip overlay may be labeled only as a supplemental
current-background/full-payout-cardinality stress run. It must not set
`production_release_authorization.scoped_pair_sufficient` to true. That flag
may change only after one of these separately reviewed acceptance paths is
complete:

1. an exact-candidate terminal-live capture at height 6,192,999 passes the
   combined-chainstate, production-RPC, restart, reorg, reindex, and compaction
   gate; or
2. a new operational background envelope states its non-consensus assumptions,
   proves the current chain remains inside them, includes production decoding
   and classification, carries an enforceable monitoring/abort threshold, and
   receives independent approval as sufficient for the stated release scope.

Until then beta and canary qualification may rely on the scoped paired gate,
while final production publication remains fail-closed.

Two possible post-alpha designs remain intentionally unimplemented:

- rolling aggregate entitlements keyed by payout script or participant; and
- deferred claim materialization with exact accrual and a canonical bounded
  drain.

Either design changes persisted semantics and requires a separate economic,
replay, wallet, and activation review. Neither is needed to enforce the bounds
above.

## Candidate observations

These observations were collected from the local mainnet v30.1.0 chainstate
and a read-only JSON-RPC scan on 2026-07-13. They are not consensus constants:

- the authenticated height-5,945,000 snapshot contains 687 whitelist scripts;
- blocks 5,950,000 through 5,952,052 contained three `QQSIGNAL` payloads at
  heights 5,950,349, 5,950,517, and 5,951,597;
- those payloads represented two distinct legacy targets; and
- replay logs showed a maximum of two simultaneously paid active signalers in
  that observed interval.

On an Apple-silicon local build from the candidate worktree, the dedicated
1,088-lookalike unit test completed in approximately 2.0 ms, and the
then-current `shadow_tests` suite completed in approximately 34.0 seconds. The
lookalike test covers every internal marker tag, including 64 ordinary outputs
per tag, and proves that collection and purge authenticate and remove none of
them. The same suite proves the 687 + 64 mainnet bound, the zero-whitelist
64-claim bound, the legacy boundary behavior, exact claim-count enumeration,
the canonical PoW winner plus 63 loser reimbursements, replay, and undo.

Those timings are candidate developer evidence only. The build used the host
liboqs, omitted Berkeley DB, and was not produced by the pinned release matrix.

The bounded-claim fixtures additionally construct blocks within one serialized
output of the contextual maximum weight. One fixture exercises both a single
multi-note transaction and thousands of independently selected candidates,
requires no more than 64 detailed accounting objects, and imposes a 120-second
evaluation/connect ceiling. A live in-memory shadowindex fixture connects a
near-maximum-weight block containing tens of thousands of malformed proof
notes, verifies a serialized block-record ceiling of 4,096 bytes and zero
detail rows, disconnects it, connects an alternate branch, reorganizes back,
and rebuilds a fresh index. Every connect, append, undo, reorg, and rebuild
phase has an independent 120-second ceiling. The exact note count is
fixture-derived and may change if unrelated block-template serialization
changes; the asserted weight, row-count, serialized-state, and time ceilings
are normative.

## External benchmark evidence

`QuantumSyntheticStateApplyUndoMaxMarkers` constructs the protocol-pinned
height-5,945,000 snapshot through `ApplyLegacyWhitelistSnapshot` and refuses to
run unless all 687 independently constructed legacy targets are present in the
snapshot. It then applies 687 qualifying PoS solve blocks, authenticates 687
active payout signals, adds the 64-proof PoW evaluation maximum, and verifies
that the measured block creates exactly 751 claim records, including exactly
64 PoW records. Every timed transition calls the production apply, inventory
checkpoint, undo, and inventory rewind paths.

Each timed epoch uses a new child UTXO cache so prior marker or tombstone
allocations cannot warm later epochs. A separate preflight repeats the same
transition four times in one persistent cache and fails if its entry count or
dynamic memory grows after the first fixed point. Preflight and every measured
iteration authenticate the pool, all 687 active signals, the inventory tip and
totals, and the complete parent-state identity after undo.

On the local Apple-silicon host, the default candidate build measured 62.814 ms
median and 64.024 ms maximum over 11 epochs. A second build with `DEBUG`,
`DEBUG_LOCKORDER`, and `DEBUG_LOCKCONTENTION` measured 106.841 ms median and
110.100 ms maximum. The evidence generator rejects any individual epoch above
2.0 seconds, not merely a slow median. That ceiling is less than 1/32 of the
64-second target spacing and was 18.2 times the slower local profile's observed
maximum.

The benchmark and its `synthetic-state` domain are mandatory in the Linux
x86-64 and native macOS evidence jobs. The generator binds the raw result and
executable to an exact clean `Blackcoin-Dev/Blackcoin` checkout and fails on a
missing domain, wrong repository or commit, dirty checkout, changed unit or
batch, irreproducible summary, or over-threshold epoch. The local measurements
above used the host liboqs and are developer evidence, not a substitute for
pinned Linux x86-64, macOS arm64/x86-64, supported Windows, sanitizer, replay,
and reorg evidence required before issue #13 is verified for production.

## Paired scoped physical LevelDB qualification

The in-memory maximum-block benchmark cannot establish physical database
growth, and live mainnet has not yet reached height 6,192,999. The Linux-only
production gate therefore keeps two claims separate: a deterministic
completed-epoch incremental shadow-family workload that is runnable now, and a
recent live-mainnet partial-epoch replay of the combined chainstate. Neither is
mislabeled as the other. Their immutable schema-2 contract is
`ci/release/shadow_resource_production_contract.json`.

The deterministic fixture writes the exact 243,000-block schedule into the
bundled LevelDB implementation: 179,771,400 claims, 539,314,200 claim-family
records, 541,701,000 total records, and 103,622,484,600 logical batch-payload
bytes after adding 199,800 maximum-size QQPROOFS buckets. Keys are
collision-free but hash-ordered. Values and batches use the
pinned serialized sizes from the production record families. The fixture
applies one atomic block at a time and persists its source-bound phase and
height. The production runner never treats a resumed suffix as a full
measurement: an interrupted apply is reset to empty, an interrupted undo or
reapply is restored from its byte-exact closed-DB predecessor and rerun in
full, and interrupted compaction is restored and measured again from its exact
pre-compaction layout. Immutable SSTables are retained with same-filesystem
hard links; mutable WAL, manifest, and metadata files are copied and fsynced.
Read-only scan, authentication, and clean-open phases also retain exact
closed-DB predecessors because opening LevelDB can perform WAL recovery or
obsolete-file cleanup. Short synthetic opens wait behind a start barrier until
their live child RSS has been sampled.
An inherited nonblocking work-directory lock prevents a retry from deleting a
database still owned by a surviving child. Completed, source-bound phases may
be reused across workflow restarts. These protections preserve database and
phase completeness. They do not evict the operating-system page cache, so
wall-time results are observed host/page-cache qualification measurements, not
universal cold-cache bounds.

The synthetic gate performs and journals:

1. the complete exact apply and a sequential record/byte scan;
2. the fixture's I/O-shaped access pattern: two complete
   541,701,000-record scans,
   179,771,400 random payout-provenance seeks, and 179,771,400 random
   demurrage-state seeks, plus 12,785,184 bounded-window QQPROOFS bucket
   seeks across the post-boundary epoch;
3. three clean opens;
4. exact removal of all 243,000 blocks and a zero-record scan;
5. exact reapplication and a second complete scan; and
6. forced compaction followed by a third complete scan.

The authentication phase must report exactly 1,083,402,000 sequential records
and 372,327,984 successful point lookups. Its raw LevelDB fixture path has a
one-day wall-time budget.
The demurrage half uses deterministic conservative-sized hit records to measure
seek cost; the 10,000-record unit test separately executes the real latest-state
decoder and verifies constant memory.
A linear-only scan or a scaled projection cannot pass as this completed-epoch
shadow-family fixture. The synthetic path does not include pre-existing or
ordinary base UTXOs, ordinary or migration quantum outputs, production Coin
and marker decoding/classification CPU, or the physical effects of those
records in a terminal combined chainstate. It is therefore not a bound on the
full terminal `getcirculatingsupply` access cardinality or wall time and is not
evidence that live mainnet has reached the final Gold Rush height.

The second runner consumes a checksum-pinned mainnet archive captured within
seven days of both the protected run and release verification. Its schema-2
manifest binds the clean candidate SHA, archive, current tip hash and height,
first and pre-Gold-Rush hashes, exact issued/spent/unspent claim counts, and a
protected-operator capture attestation. The attached capture RPC transcript
must report mainnet, equal block/header heights, the archived best-block hash,
no initial download, and at least one peer. These checks authenticate internal
consistency and operator provenance; the offline gate does not independently
query the global network for the tip. The operator records that transcript
while connected, stops the exact candidate cleanly without advancing the
datadir, and only then archives it. The tip must be between heights
5,950,000 and 6,192,999. The runner rejects stale or future-dated manifests,
scaled or inconsistent counts, unsafe or symlinked paths, a wrong repository
or dirty tree, changed binaries, or any source/contract/archive digest
mismatch.

Against that current live partial epoch, `blackcoind` performs a full
`-reindex-chainstate` replay, authenticated `getcirculatingsupply`, three clean
opens, invalidation to height 5,949,999, reconsideration to the captured tip,
and forced compaction. Every phase is anchored to the manifest's exact block
hash and finishes with a clean daemon shutdown. Undo and reapply are separately
journaled so each four-day maximum fits inside the five-day workflow window.
Mutation, lifecycle-scan, and the complete three-open startup sequence retain
and restore chainstate plus block-index physical predecessors before a retry.
When live mainnet eventually reaches 6,192,999,
the same contract requires completed-live equivalence. The protected operator
must then provide the terminal capture; the offline runner cannot infer that
the global chain has reached the terminal height. That future confirmation
does not block measuring the complete synthetic shadow-family envelope now.

A nominal five-second sampler records peak process RSS and physical database
bytes by SSTable, WAL, manifest, and other file families. Periodic samples tolerate an
ordinary LevelDB file disappearing during rename/delete churn; closed-database
predecessors and endpoints remain strict. Each phase records its actual sample
count, longest sweep, and longest start-to-start gap; a sweep above one second
or a gap above seven seconds fails. The artifact records kernel, CPU, RAM,
filesystem/mount/device, compiler, Python, and page-cache policy. The paired
verifier requires an exact clean `Blackcoin-Dev/Blackcoin` origin and SHA,
hashes and semantically
pins the contracted schedule, serializer, cache-allocation, state-transition,
runner, and workflow sources, authenticates the fixture and all three binaries,
recomputes ratios and summary peaks, and rejects altered phase or cardinality
labels.

The sampled/endpoint maximum is explicitly an observed value, not an upper
bound on create-and-delete overlap entirely between samples. It may trigger a
compaction requirement, but it cannot prove that an unseen transient was
smaller.

Authenticated compaction is required if steady physical bytes exceed 2.0 times
retained logical payload, maximum observed sampled/endpoint bytes exceed 3.0
times that payload, obsolete files exceed 0.10 times it, or forced compaction
reclaims at least ten
percent of the steady database. The gate also rejects WAL above 4 GiB, peak RSS
above 16 GiB, a clean open above ten minutes, exact authentication or a scan
above one day, or apply/undo/reapply/compaction above four days. These are
qualification budgets on the recorded host and cache policy, not consensus or
universal hardware limits.

No authenticated deletion protocol is implemented in this candidate. If the
measurements require one, verification fails. The protected
`v30.1.1 paired scoped shadow resource gate` uploads the synthetic
evidence, current-live evidence and fixture manifest, contract, fixture binary,
daemon, and CLI. The release workflow accepts only an unexpired successful
artifact from that workflow and the exact release SHA, then independently
re-verifies both evidence halves.

The source, runner, and workflow are not themselves physical evidence. A green
paired artifact qualifies only the scopes recorded in its machine-readable
contract. Issue #13 remains Partial after that artifact until terminal-live
combined-chainstate and production-RPC evidence, or a separately reviewed
conservative equivalent, exists. No scoped measurement may be presented as a
universal or terminal production bound.
The contract sets scoped-pair sufficiency to false for final production
publication. Beta/canary testing may use the scoped evidence, but the release
workflow cannot publish a final production release until the remaining
terminal qualification is reviewed and the authorization is explicitly
changed.
