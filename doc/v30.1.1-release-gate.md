# Blackcoin Core v30.1.1 production gate

v30.1.1 is not production-ready merely because it compiles or an individual
test passes. The release commit must satisfy every P0 and P1 item in the master
roadmap, preserve the designated legacy base chain during Gold Rush, reproduce
the historical shadow ledger, and produce independently verifiable artifacts.

## v30.1.1-alpha.1 scope freeze

`v30.1.1-alpha.1` is an urgent prerelease for operator testing. It is not the
production v30.1.1 release and must not be represented as having passed the
production gate below. The alpha scope is frozen to the fixes needed to make
v30.1.0 deployments usable and to avoid catastrophic upgrade failures:

- authenticated chainstate and shadow-replay reconstruction, exact marker
  provenance, fail-closed rebuild/reindex handling, and preservation of base
  chain validity;
- bounded live-output staking selection and status work so staking and the Qt
  wallet remain responsive;
- the already-integrated wallet lifecycle and durability, deterministic shadow
  state, Gold Rush PoW claim accounting, and payout-reconciliation fixes; and
- issue #5 light/dark theme readability, including the automated palette,
  contrast, and theme-transition regressions already implemented for the alpha.

An alpha candidate is blocked by a regression that can corrupt or erase
chainstate, make replay nondeterministic, fork base or shadow accounting, lose
wallet keys, or leave staking or the primary GUI paths unusable. Other hazards
remain on the production roadmap and do not expand this alpha's scope.

The alpha now uses a staged, journaled chainstate reconstruction instead of a
destructive preflight-and-wipe sequence. The source databases remain in named
backups while stored block bodies, transaction Merkle data, witness
commitments, snapshot metadata, saved tips, and authenticated auxiliary state
are rebuilt. An interrupted or failed build rolls back. A successful build
enters `COMMIT_READY`; only a separate process that reopens and verifies the
replacement may retire the source backups. Dedicated unit and functional tests
cover missing, truncated, and mutated block records; malformed or unknown
snapshot bases; null or interrupted saved tips; snapshot-only sources;
shutdown; repeated reindex flags; damaged journal topology; restart; reorg;
and deterministic Gold Rush reconstruction. Those tests are candidate evidence
until they pass on the final exact alpha commit.

The following non-catastrophic hardening is deferred from alpha.1 but remains
required work before the production release:

### Alpha artifact gate and upgrade warning

A manual `workflow_dispatch` build is initially an unpublished alpha artifact
build. It does not begin its platform matrix until that exact commit passes source-policy
and lint checks, the pinned DEBUG/lock-order unit suite, the critical protocol
and wallet functional suite, and the real v26.2.0, v28.4.0, v30.1.0, and
candidate mixed-version gate. The open pull request continues running its full
pull-request gate independently for the same commit. Manual alpha builds skip
only the production-only native macOS vectors, complete functional/soak suite,
sanitizer matrix, and pinned-corpus fuzz smoke. After exact-byte operator
verification, the unchanged bundle may be published at `v30.1.1-alpha1` only
as a GitHub prerelease with `latest=false`, immutable assets, full-SHA names,
unsigned checksums, and the `UNSIGNED-CANARY` warning intact. That optional
promotion is not a production release. A signed `v30.1.1` tag push sets
`release_mode=true` and must pass all of those production jobs before any
publication or signing job can run.

The supported alpha upgrade path is an official v30.1.0 wallet and datadir, or
a clean wallet created by the alpha. A wallet file that was actually opened by
an unpublished intermediate source build which retroactively enabled the
competing-claim reimbursement rule could retain synthetic payout records that
the corrected historical replay no longer recognizes. Preserve such a wallet
and do not rely on its displayed synthetic balance until wallet reconciliation
lands; use a pre-intermediate backup only after separately verifying that it
contains every current legacy and quantum key. This condition does not affect
direct upgrades from the official v30.1.0 release. The unsafe all-platform
workflow run was canceled before it built or uploaded any artifact.

## v30.1.1 beta gate

A beta is not an alpha relabel. The candidate branch must first contain every
P0/P1 implementation assigned to that beta and a disposition for every
remaining roadmap condition. A manual `30.1.1-betaN` build then requests the
complete exact-SHA safety matrix: the alpha core gate plus exhaustive
functional/soak coverage, native macOS cryptographic vectors,
address/undefined/thread sanitizers, and the pinned fuzz corpus. The platform
matrix remains blocked until those jobs pass.

Beta artifacts remain unsigned, unnotarized, and initially unpublished. They
retain the full source SHA and `UNSIGNED-CANARY` warning and require the same
isolated two-start upgrade and rollback rehearsal as alpha. A successful beta
gate supplies test evidence; it does not satisfy production signing,
independent review, operator canary, or final-publication requirements.

### Recovery hardening candidate

The beta candidate uses an explicit no-replace rename for first-run datadir
migration and journaled chainstate moves. Windows uses a native write-through
move without replacement semantics; macOS and Linux use their atomic
exclusive-rename primitives. An existing destination remains authoritative and
is never overwritten by the move.

`ChainstateManager` also records successful completion of
`VerifyLoadedChainstate` for the current load. A `COMMIT_READY` rebuild cannot
advance to backup cleanup without that proof, and any later failed or
interrupted verification clears it. The rebuilding process may still make the
required `BUILDING` to `COMMIT_READY` transition without retiring its source.

Dedicated tests cover existing-destination preservation, interruption while
staging and building, `COMMIT_READY` restart verification, `CLEANUP_READY`
restart, failed and interrupted verification, backup retention,
disconnect/reconnect, and reindex. The recovery suite now pauses on the
production path immediately after every durable chainstate journal transition
(`PREPARED`, `BUILDING`, `ROLLING_BACK`, `COMMIT_READY`, and `CLEANUP_READY`),
terminates the daemon with the operating system, and proves deterministic
recovery in another process. The first-run datadir suite does the same after a
staged import, recovery record, active-directory move, and promotion. A
source-size preflight fails before the journal or any rename when the existing
chainstate plus the 50 MiB reserve will not fit. POSIX testing additionally
applies a real process file-size quota after durable `BUILDING`, forces the
replacement LevelDB write path to fail, and proves restart restoration. The
wallet gate restores both database formats and proves a legacy key still signs
while every restored quantum key passes a fresh ML-DSA challenge during a
second verified backup.

Production-network startup rejects nonzero `-prune`; beta operators remain
restricted to archival `prune=0`. Issue #10 remains Partial until the final
exact-beta matrix reproduces the kill and write-exhaustion results on its exact
binaries, supplies native Windows migration/power-loss evidence, and completes
the documented two-start rollback rehearsal on release artifacts. That
rehearsal must compare legacy and quantum key inventories, require the verified
backup's sign-every-key check, and prove that no post-upgrade wallet state is
silently reused or lost when the cold pre-upgrade copy is restored. Production
pruning is not a beta feature and cannot be claimed from regtest coverage.

## Status definitions

- **Not started:** no scoped implementation or release evidence exists.
- **Partial:** some implementation or tests exist, but an acceptance condition
  is missing or a relevant test is failing.
- **Candidate:** the implementation and dedicated regression tests exist on the
  proposed release commit, but the complete release matrix has not passed.
- **Verified:** the release commit passed its dedicated tests, adversarial
  review, historical replay, relevant mixed-version checks, and documentation
  review. Only a verified issue may be closed for the release.

## Mandatory test sequence

1. Perform clean, pinned Linux and macOS builds with Berkeley DB, SQLite, and a
   separate Qt-enabled build. Run the complete unit suite.
2. Run all Quantum Quasar functional tests serially, including activation,
   Gold Rush PoW/PoS, migration, demurrage, value paths, replay, reorg, wallet,
   raw transaction, and integration-soak coverage.
3. Run concurrent two-wallet signalling and staking stress with repeated RPC
   calls, block connection, reorg, and shutdown under lock-order diagnostics
   and the thread sanitizer.
4. Run bidirectional block production and acceptance using exact pinned
   v26.2.0, v28.4.0, v30.1.0, and proposed v30.1.1 binaries.
5. Replay from height 5,944,999 or earlier and compare base hashes, synthetic
   credits, pools, active signals, counts, maturity, and accounting before and
   after restart, reindex, disconnect/reconnect, and restore.
6. Run forced-shutdown, datadir-migration, disk-space, wallet backup/restore,
   and rollback rehearsals. Before production, separately restore and audit the
   disabled Blackcoin PoS pruning engine, then run a real archival-to-pruned-to-
   full-redownload replay comparison; alpha and beta must not claim pruning
   support until that gate passes.
7. Run lint, fuzz smoke, address/undefined/thread sanitizers, cryptographic
   vectors, resource benchmarks, and dependency/advisory review.
8. Build the complete platform matrix from pinned dependencies. Two independent
   builders must reproduce matching artifacts. Sign the source commit, tag,
   checksums, manifests, SBOM, and platform packages.

### Pinned script-assets gate

The unit and sanitizer jobs provision a checksum-pinned
`script_assets_test.json` and set both `DIR_UNIT_TEST_DATA` and
`REQUIRE_SCRIPT_ASSETS_TEST=1`. A missing directory, missing file, checksum
mismatch, malformed transaction, or changed classification count fails the
job. The gate must not treat an unset data path as a passing test.

The reviewed upstream corpus contains 2,244 vectors. Bitcoin version-1
transactions omit the `nTime` field that Blackcoin serializes for transaction
versions below 2. The gate therefore reads the initial signed little-endian
`int32` before transaction deserialization, explicitly excludes the 1,161
vectors whose version is below 2, and runs all 1,083 vectors whose version is
2 or greater. A corpus update must review and update the source commit,
checksum, and all three counts together. It must not deserialize Bitcoin
version-1 bytes as a Blackcoin version-1 transaction.

The fuzz runner also executes a source-pinned 89-byte `block` regression before
the external corpus. Its SHA256 is
`4cc1cc0892fd66dbde05b5b75332caab5f158f31068d368c82bca03b8802b936`.
The input reaches `CheckBlock`, so the target must construct and retain a
complete regtest `TestingSetup` with an active chainstate. A missing setup,
changed regression input, omitted `block` target, sanitizer finding, or nonzero
target exit fails the release fuzz job before the upstream corpus can be
treated as passing. Ordinary developer runs that intentionally select another
single fuzz target remain available without the release-only requirement flag.

The current resource-evidence schema records the measured 64-evaluation
Argon2 block cap, 8,215-verification ML-DSA block cap, and production-path
validation of an 8,198-input quantum block at 31,997,596 weight units. The
`crypto`, `large-block`, and `synthetic-state` domains are mandatory for every
pull request and candidate. The synthetic benchmark authenticates the
height-5,945,000, 687-entry snapshot bound, exercises all 751 maximum claims,
and fails if apply/undo changes parent identity, grows persistent cache state,
or any measured epoch exceeds 2.0 seconds. Unsigned alpha canaries do not claim
the still-required pinned cross-platform production evidence.

## Production signing configuration

Manual workflow runs build and compare unsigned candidates but does not itself
publish. A separately authorized public alpha promotion follows the prerelease
controls above and does not weaken this production path.
The signed `v30.1.1` tag path fails closed unless the protected
`production-release` environment provides all of the following:

- `RELEASE_GPG_PUBLIC_KEY_B64`, `RELEASE_GPG_PRIVATE_KEY_B64`, and the exact
  `RELEASE_GPG_FINGERPRINT` repository variable;
- `WINDOWS_CERTIFICATE_PFX_B64`, `WINDOWS_CERTIFICATE_PASSWORD`, the exact
  `WINDOWS_CERTIFICATE_SHA256`, and an RFC 3161 `WINDOWS_TIMESTAMP_URL`;
- `MACOS_CERTIFICATE_P12_B64`, `MACOS_CERTIFICATE_PASSWORD`, the exact
  `MACOS_CERTIFICATE_SHA256`, and `MACOS_SIGNING_IDENTITY`; and
- `APPLE_TEAM_ID`, `APPLE_NOTARY_KEY_ID`, `APPLE_NOTARY_ISSUER_ID`, and
  `APPLE_NOTARY_PRIVATE_KEY_B64`.

The production path verifies the signed source commit and signed annotated tag,
requires byte-identical artifacts from two isolated builds, Authenticode-signs
Windows executables, Developer-ID-signs and notarizes macOS packages, emits an
SPDX SBOM and in-toto provenance statement, and signs the provenance and final
checksum manifest. It also publishes GitHub build-provenance and SBOM
attestations. Repository-level immutable releases must be enabled before the
tag is pushed; the workflow refuses to overwrite any existing release.

Before the release-candidate commit is approved, repository administrators
must also restrict the `production-release` environment to the `v30.1.1` tag,
require an independent human reviewer, prevent administrator bypass, and make
the complete pull-request safety gate mandatory on the protected branch. No
credential is copied into a workflow file, repository variable, build log, or
release asset.

## Publication rule

Pull requests may be opened as drafts for review. A draft branch or successful
single-platform build is not a release candidate. The release workflow must
publish only the exact merge commit that passed every required check. The final
release flag and final tag may be set only on that commit.
