# Blackcoin Core v30.1.1 production gate

v30.1.1 is not production-ready merely because it compiles or an individual
test passes. The release commit must satisfy every P0 and P1 item in the master
roadmap, preserve the designated legacy base chain during Gold Rush, reproduce
the historical shadow ledger, and produce independently verifiable artifacts.

## Status definitions

- **Not started:** no scoped implementation or release evidence exists.
- **Partial:** some implementation or tests exist, but an acceptance condition
  is missing or a relevant test is failing.
- **Candidate:** the implementation and dedicated regression tests exist on the
  proposed release commit, but the complete release matrix has not passed.
- **Verified:** the release commit passed its dedicated tests, adversarial
  review, historical replay, relevant mixed-version checks, and documentation
  review. Only a verified issue may be closed for the release.

## Mandatory test sequence

1. Perform clean, pinned Linux and macOS builds with Berkeley DB, SQLite, and a
   separate Qt-enabled build. Run the complete unit suite.
2. Run all Quantum Quasar functional tests serially, including activation,
   Gold Rush PoW/PoS, migration, demurrage, value paths, replay, reorg, wallet,
   raw transaction, and integration-soak coverage.
3. Run concurrent two-wallet signalling and staking stress with repeated RPC
   calls, block connection, reorg, and shutdown under lock-order diagnostics
   and the thread sanitizer.
4. Run bidirectional block production and acceptance using exact pinned
   v26.2.0, v28.4.0, v30.1.0, and proposed v30.1.1 binaries.
5. Replay from height 5,944,999 or earlier and compare base hashes, synthetic
   credits, pools, active signals, counts, maturity, and accounting before and
   after restart, reindex, disconnect/reconnect, and restore.
6. Run pruned-node, forced-shutdown, datadir-migration, disk-space, wallet
   backup/restore, and rollback rehearsals.
7. Run lint, fuzz smoke, address/undefined/thread sanitizers, cryptographic
   vectors, resource benchmarks, and dependency/advisory review.
8. Build the complete platform matrix from pinned dependencies. Two independent
   builders must reproduce matching artifacts. Sign the source commit, tag,
   checksums, manifests, SBOM, and platform packages.

## Production signing configuration

Manual workflow runs build and compare unsigned candidates but never publish.
The signed `v30.1.1` tag path fails closed unless the protected
`production-release` environment provides all of the following:

- `RELEASE_GPG_PUBLIC_KEY_B64`, `RELEASE_GPG_PRIVATE_KEY_B64`, and the exact
  `RELEASE_GPG_FINGERPRINT` repository variable;
- `WINDOWS_CERTIFICATE_PFX_B64`, `WINDOWS_CERTIFICATE_PASSWORD`, the exact
  `WINDOWS_CERTIFICATE_SHA256`, and an RFC 3161 `WINDOWS_TIMESTAMP_URL`;
- `MACOS_CERTIFICATE_P12_B64`, `MACOS_CERTIFICATE_PASSWORD`, the exact
  `MACOS_CERTIFICATE_SHA256`, and `MACOS_SIGNING_IDENTITY`; and
- `APPLE_TEAM_ID`, `APPLE_NOTARY_KEY_ID`, `APPLE_NOTARY_ISSUER_ID`, and
  `APPLE_NOTARY_PRIVATE_KEY_B64`.

The production path verifies the signed source commit and signed annotated tag,
requires byte-identical artifacts from two isolated builds, Authenticode-signs
Windows executables, Developer-ID-signs and notarizes macOS packages, emits an
SPDX SBOM and in-toto provenance statement, and signs the provenance and final
checksum manifest. It also publishes GitHub build-provenance and SBOM
attestations. Repository-level immutable releases must be enabled before the
tag is pushed; the workflow refuses to overwrite any existing release.

Before the release-candidate commit is approved, repository administrators
must also restrict the `production-release` environment to the `v30.1.1` tag,
require an independent human reviewer, prevent administrator bypass, and make
the complete pull-request safety gate mandatory on the protected branch. No
credential is copied into a workflow file, repository variable, build log, or
release asset.

## Publication rule

Pull requests may be opened as drafts for review. A draft branch or successful
single-platform build is not a release candidate. The release workflow must
publish only the exact merge commit that passed every required check. The final
release flag and final tag may be set only on that commit.
