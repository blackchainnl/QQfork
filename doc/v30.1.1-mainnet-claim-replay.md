# v30.1.1 mainnet competing-claim replay

## Status and scope

This report records a read-only replay of the public Blackcoin main chain from
height 5,950,000 through a fixed tip at height 5,951,657. It compares the
v30.1.0 transaction-order rule with the canonical-rank and fee-reimbursement
rule implemented by candidate source snapshot
`997d80ba7fe8411e7c76f98422496edddea92b02`.

The replay establishes the historical allocation effect of activating that
candidate rule at height 5,950,000. It does not validate a later activation
height or a later source revision.

## Result

The replay found 103 blocks containing more than one proof-valid,
input-matched Gold Rush PoW claim. Under v30.1.0, the first such claim in base
block transaction order received the entire accumulated PoW pool. The other
110 valid claims paid base-chain fees but received no shadow credit.

The candidate canonical-rank rule preserved the fixed pool in every replayed
block and reimbursed valid losing claims without increasing total shadow
issuance. It selected a different winner in 57 of the 103 competing-claim
blocks, however. Activating that rule at height 5,950,000 would therefore
rewrite historical claimant allocation in those 57 blocks.

That allocation change is the principal release risk. The aggregate fee
reimbursement was 0.031434 BLK. The aggregate amount transferred away from the
v30.1.0 winner across the 57 reranked blocks was 30,739.997164 BLK. This is a
redistribution of the unchanged fixed pools, not new issuance.

A production release must not activate this rule retroactively unless the
historical reranking is an explicit, reviewed protocol decision. A candidate
that instead preserves v30.1.0 results through a future activation boundary
requires a separate replay against its final source commit.

## Replay method

1. Resolve every height in the fixed range with `getblockhash`, then obtain the
   block, transactions, fees, and undo prevouts with `getblock <hash> 3`.
2. Treat a marker as exact only when its script is `OP_RETURN` followed by one
   pushed element beginning with the bytes `QQSPROOF`.
3. Decode the QQP2 PoW claim and require the quantum payout shape used by the
   audited source.
4. Reproduce `RetargetedBits` and the vendored Argon2id proof check with time
   cost 1, memory 1,024 KiB, and one lane.
5. Reproduce the audited serialization and hash operations, including source
   outpoint, proof payload, previous block hash, and height.
6. Canonicalize P2PK spent outputs to P2PKH where required and verify that the
   claim input spends the encoded source script using undo prevouts.
7. Compute each actual base fee as spent prevout value minus transaction output
   value and compare it with the fee reported by the node.
8. Track the scheduled accumulated PoW pool at every height. Reset the pool
   whenever a block contains a valid winning claim.
9. Model the v30.1.0 winner as the first valid claim in block transaction
   order.
10. Model the candidate winner as the lowest canonical rank, with the audited
    source's txid, vout, and proof-payload collision tie-breakers. Reimburse
    each valid loser by `min(actual fee, 0.01 BLK)` and give the winner the
    fixed-pool remainder.

The exact candidate source snapshot, rather than an independently designed
claim rule, defined every classification and amount in this replay.

## Independent reconciliation

Retained v30.1.0 logs from 30 independently running node instances contained
exactly 103 unique `ignored duplicate QQPROOF claims at height` observations.
Their ordered height sequence matched the replay's 103 competing-claim heights
and the height-sequence SHA-256 recorded above.

At height 5,951,655, the live v30.1.0 state reported 767 paid PoW claim blocks,
the last paid claim at height 5,951,650, target bits 12, and an accumulated PoW
pool of 1,450 BLK. The offline replay matched all four values at that same
height. This checkpoint is separate from the fixed-tip height 5,951,657 and is
not represented as an atomic live-state comparison at the later tip.

## CSV field interpretation

The CSV contains one row per competing-claim block. Amounts are integer
satoshis. `legacy_winner_credit_delta_sat`,
`replacement_winner_credit_delta_sat`, and
`remaining_claimants_credit_delta_sat` are disjoint claimant groups; their sum
is `block_total_credit_delta_sat`. The latter is zero in every row.

When the winner is unchanged, `value_reassigned_sat` is the reimbursement
removed from that winner's pool credit and credited to the losers. When the
winner changes, it is the amount removed from the v30.1.0 winner after that
claim receives its candidate reimbursement. The delta columns and
`value_reassigned_sat` describe the same allocation change from different
perspectives and must not be added together.

## Public reproduction checks

The following shell outline fixes the extraction range and verifies the public
block source. `$CLI` denotes a locally configured `blackcoin-cli` command.

```sh
START=5950000
END=5951657
EXPECTED=ddd881d477487e9becf1d5cf3accd1971f8b641cbd8ac60071064ddfee09dd97

test "$($CLI getblockhash "$END")" = "$EXPECTED"
$CLI getindexinfo

for h in $(seq "$START" "$END"); do
    hash=$($CLI getblockhash "$h")
    $CLI getblock "$hash" 3
done

sha256sum doc/audits/v30.1.1-mainnet-competing-claims.csv
wc -l doc/audits/v30.1.1-mainnet-competing-claims.csv
tail -n +2 doc/audits/v30.1.1-mainnet-competing-claims.csv \
    | cut -d, -f1 | paste -sd, - | tr -d '\n' | sha256sum
```

Block extraction alone does not validate QQP2 proofs. A full reproduction must
use the exact audited proof parameters, serialization, source-input checks,
fee reconstruction, pool schedule, and canonical ordering described above.

## Remaining release evidence

This independent replay does not replace an authoritative replay by the final
candidate binary. Before a production tag, run that exact binary from a clean
state with `-reindex-chainstate`, compare its shadow index and
`getshadowblock` results row-for-row at a fixed tip, restart it, and repeat the
comparison. Repeat the mainnet allocation audit if the activation boundary,
ranking inputs, reimbursement rule, proof validation, pool schedule, replay
schema, or source commit changes.

Generating exact-SHA alpha artifacts does not require that production replay.
One offline canary on a copied datadir may perform it as the alpha acceptance
test. The replay and clean-restart comparison must pass before the alpha is
rolled out to the mainnet fleet, and they remain mandatory before any
production tag.
