# Blackcoin v30.1.1 dependency, provenance, and legacy-delta audit

Status date: 2026-07-14

This document records the evidence reviewed for the v30.1.1 dependency and
network-capability work. It is not a statement that the release is production
ready. The release remains gated on the platform builds and unresolved legacy
deltas identified below.

## Scope and immutable baselines

The audit used these immutable revisions:

* v30.1.1 audit starting point:
  `7fde566f5f564e43f4c2d76502bd7e43c819e35f`
* v30.1.0 release baseline:
  `f647dc75c9479c03e81414f145a8d233b60959c7`
* active legacy repository baseline:
  `5c1a4a5efc3886a69f01d50109c251d153ed092e`

The repositories do not have a common git object ancestor. The v30 history was
squashed before this audit. Commit presence therefore cannot be inferred from
`git merge-base`, and no legacy commit was bulk cherry-picked. Relevant changes
were compared by path and behavior because the v30 code also reserves witness
version 16 for Quantum Quasar and implements a staged shadow-ledger schedule.

The machine-readable source inventory, recipe digests, dispositions, and
provenance counts are in
`ci/release/dependency-security-manifest.json`. The pull-request gate runs
`ci/release/verify_dependency_security.py` and fails if that evidence or the
reviewed source changes without a corresponding review update.

### Vendored crc32c ARM64 sanitizer backport

The vendored crc32c baseline used direct `uint16_t`, `uint32_t`, and
`uint64_t` pointer casts in its ARM64 hardware path. LevelDB may supply a byte
slice that is not naturally aligned, so native ARM64 UBSan stopped in
`crc32c::ExtendArm64` while `TestingSetup` opened the block-index database. The
stack had not entered Blackcoin's `CheckBlock` path.

v30.1.1 backports Google crc32c commit
[`d3d60ac6e0f16780bcfcc825385e1d338801a558`](https://github.com/google/crc32c/commit/d3d60ac6e0f16780bcfcc825385e1d338801a558).
That upstream change replaces the invalid pointer casts with the library's
`memcpy`-backed little-endian loads. It does not disable the alignment
sanitizer, select the portable CRC path, or change the CRC32C result. The
manifest pins the three touched vendored files. A direct unaligned-input vector
and the pinned block-fuzz regression run under native ARM64 UBSan.

The manifest compares all 31 candidate recipes against both baselines, not only
the seven version updates in the table. That complete comparison exposed two
additional active-legacy differences retained from v30.1.0:

* libnatpmp remains at `07004b97cf691774efebe70404cf22201e4d330d`;
  active legacy uses `f2433bec24ca3d3f22a8a7840728a3ac177f94ba`.
  The [upstream comparison](https://github.com/miniupnp/libnatpmp/compare/07004b97cf691774efebe70404cf22201e4d330d...f2433bec24ca3d3f22a8a7840728a3ac177f94ba)
  adds response-type retry handling, while the active-legacy recipe changes the
  deterministic build from Make to CMake. NAT-PMP is optional and
  non-consensus. The update is deferred until its Linux, Windows, and macOS
  build and behavior matrix passes.
* Native and target libmultiprocess remain at
  `1af83d15239ccfa7e47b8764029320953dd7fdf1`; active legacy uses
  `6aca5f389bacf2942394b8738bbe15d6c9edfb9b`. The
  [upstream comparison](https://github.com/bitcoin-core/libmultiprocess/compare/1af83d15239ccfa7e47b8764029320953dd7fdf1...6aca5f389bacf2942394b8738bbe15d6c9edfb9b)
  is coupled to Cap'n Proto 1.0.1 and changes generated IPC code and shutdown
  ordering.
  Multiprocess executables are experimental and disabled by default. A partial
  update could create an ABI mismatch or shutdown deadlock, so the pair remains
  deferred to the multiprocess compatibility matrix.

The v30-line-only liboqs recipe is required by Quantum Quasar. The three
v30-line-only native Darwin tool recipes support cross-compilation and are
unchanged from v30.1.0. The manifest records active legacy as absent for each;
it does not silently treat absence as an equal version.

The manifest resolves every package, including target/native aliases and Qt's
three source archives, to an HTTPS URL and SHA-256. It pins the digest of every
depends recipe. The native libmultiprocess recipe now names the current
`bitcoin-core/libmultiprocess` repository instead of relying on GitHub's
redirect from the former owner. The archive hash is unchanged. The SPDX
generator consumes this manifest rather than emitting `NOASSERTION` for aliased
or extra sources.

The QRencode 4.1.1 URL formerly published at fukuchi.org now returns 404. The
depends framework therefore retrieves the same release archive from its
configured Bitcoin Core fallback cache. The manifest records that reachable
cache URL and the recipe's unchanged SHA-256; the cached bytes were verified
against that digest and include the release-generated `configure` script.

The consensus-critical liboqs pin remains 0.15.0. Upstream has published
advisories affecting HQC, Kyber, and XMSS, including
[CVE-2026-44518](https://github.com/open-quantum-safe/liboqs/security/advisories/GHSA-wf7v-fhxj-73m2)
and
[CVE-2026-46344](https://github.com/open-quantum-safe/liboqs/security/advisories/GHSA-2wxh-55qf-c7wg).
Those algorithms are not compiled. The recipe uses
`OQS_MINIMAL_BUILD=SIG_ml_dsa_44`, disables OpenSSL and distribution dispatch,
and selects generic code paths. The upstream
[0.15.0 configuration guide](https://github.com/open-quantum-safe/liboqs/blob/0.15.0/CONFIGURE.md)
defines the minimal build as including only the selected algorithms. The gate
checks these exact flags. Artifact inspection and the complete cryptographic
sanitizer matrix remain separate release gates.

### Qt residual risk and reachability

Qt 5 reached the end of its standard release stream after 5.15.19. Qt reports
[CVE-2025-14575](https://www.qt.io/blog/security-advisory-untrusted-search-path-vulnerability-in-openssl)
as affecting all Qt 5 releases, with no Qt 5 fixed release. That issue concerns
the OpenSSL TLS backend on Unix-like systems. The deterministic Qt configuration
uses `-no-openssl`, `-no-schannel`, and `-no-securetransport`, and this audit
found no first-party `QSsl*` or Qt-network TLS use. The affected backend is thus
not intended to be built or reachable in the release configuration. Release
artifacts must still be inspected to confirm the configured features and linked
libraries.

Qt also reports CVE-2025-6338 in the Schannel backend and CVE-2025-12385 in Qt
Quick. Schannel is disabled. The build fetches `qtbase`, `qttools`, and
`qttranslations`, not `qtdeclarative`. The manifest gate rejects adding the
affected backend or module without a new review.

The 2026 Qt SVG marker issue applies to Qt 6.7 and later, not Qt 5.15. The
deterministic build also does not download or link the `qtsvg` module. Source SVG
assets do not establish runtime use of that module.

## Quantum Quasar P2P capability

v30.1.1 advertises `NODE_QUANTUM_QUASAR` on service bit 24. The bit communicates
that a node understands the shadow ledger and its RPC surface. It is not an
authenticated readiness vote, a consensus activation signal, or a requirement
for peer selection. `GetDesirableServiceFlags()` remains unchanged, so a legacy
peer offering `NODE_NETWORK` remains desirable and mixed-version connectivity
is not partitioned by the new advertisement.

The unit test checks the serialized service name and legacy desirability. The
`rpc_net.py` test connects one peer offering only `NODE_NETWORK` and one peer
also offering `NODE_QUANTUM_QUASAR`, verifies their exact masks and names, and
verifies that both remain connected. The pruned-node test checks the local
advertised mask.

## Bitcoin Core security advisories after the v30 lineage

[CVE-2024-52911](https://bitcoincore.org/en/2026/05/05/disclose-cve-2024-52911/)
is a high-severity use-after-free caused when queued script checks outlive their
`PrecomputedTransactionData`. The candidate is directly mitigated:
`ConnectBlock` constructs a fully sized `txsdata` vector before
`CCheckQueueControl`. C++ reverse destruction therefore waits on the control
before freeing the pointed-to data. The gate enforces that ordering and the
preallocation rationale.

The following published low-severity issues postdate both the v30.1.0 lineage
and the active legacy 28.x branch:

* [CVE-2024-52919 part 2](https://bitcoincore.org/en/2025/04/28/disclose-cve-2024-52919/)
  requires sustained address spam for more than a year even at 1,000 attacking
  peers. The 64-bit addrman identifier change is deferred to isolated P2P tests.
* [CVE-2025-46597](https://bitcoincore.org/en/2025/10/24/disclose-cve-2025-46597/)
  concerns a pathological 32-bit configuration. Official candidate artifacts
  are 64-bit only.
* [CVE-2025-46598](https://bitcoincore.org/en/2025/10/24/disclose-cve-2025-46598/)
  spans three later script-resource mitigations. It is deferred with policy and
  consensus-result differential tests required before porting.
* [CVE-2025-54604](https://bitcoincore.org/en/2025/10/24/disclose-cve-2025-54604/)
  and [CVE-2025-54605](https://bitcoincore.org/en/2025/10/24/disclose-cve-2025-54605/)
  are long-duration log-filling issues. Global log rate limiting is deferred to
  an observability-preserving backport.

The manifest records an owner, target, and risk for every deferral. These
deferrals are not claims that the later changes are unnecessary.

## Inherited copyright provenance

`contrib/devtools/restore_inherited_copyrights.py` compares the two immutable
trees at the same path. It restores a legacy Bitcoin Core notice only when:

1. the legacy file contains exactly one such notice;
2. the current baseline contains an inherited Blackcoin-family header; and
3. the normalized non-copyright bodies are identical or have line similarity
   of at least 0.95.

Vendored subtrees, generated/data/translation files, binaries, and patch
artifacts are excluded. The resulting manifest records 664 eligible files: 306
normalized-identical and 358 at or above 95% body similarity. It records 11
excluded files and 641 ambiguous files. Of the ambiguous files, 508 are below
the threshold and 133 lack an inherited Blackcoin-family header. Ambiguous files
are not modified by the tool.

The generated manifest is
`contrib/devtools/copyright-provenance-v30.1.1.json`. The dependency-security
gate pins its complete SHA-256, immutable revisions, schema, and the counts of
664 restored, 11 excluded, and 641 ambiguous paths. It fails closed if any of
those values drift.

## Remaining production gates

The following work remains before this dependency stack can be classified as
production ready:

1. Build the complete deterministic dependency graph and release artifacts on
   Linux x86-64 and arm64, Windows x86-64, and macOS x86-64 and arm64. The Qt
   update has only completed sequential patch application in this audit.
2. Verify the feature set and linked libraries of every artifact, including the
   absence of the Qt OpenSSL backend.
3. Run wallet unit and functional coverage against artifacts linked to the new
   SQLite build rather than only the developer host build.
4. Run the complete multiprocess build after rebuilding both native and target
   Cap'n Proto components; the isolated native package build alone is not a
   sufficient release test.
5. Build and exercise the active-legacy libnatpmp update on every release
   platform before changing that pin.
6. Run an advisory scanner against the generated manifest-backed SBOM and the
   actual release artifacts. Aging platform pins, including fontconfig,
   libxkbcommon, the X11 stack, and Boost, were inventoried and pinned but were
   not established as vulnerability-free by this audit.
7. Complete the owned staking and wallet-migration gates before promoting a
   beta candidate to production. Their exact deferrals and risks remain in the
   machine-readable ledger.

Until those gates pass, the dependency/security roadmap item is a release
candidate, not complete.
